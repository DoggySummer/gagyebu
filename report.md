# 보고서 작성 기능 (기획 메모)

> 구현 전 단계: 아래 내용을 기준으로 `/report` 페이지와 사이드바를 추가할 예정이다.

## 1. 사이드바

- 위치: **「엑셀 추가하기」 항목 바로 아래**에 새 내비 항목 추가.
- 라벨: **보고서 작성하기**
- 동작: 클릭 시 **`/report`** 로 이동 (Next.js `Link` 또는 동일한 패턴으로 기존 메뉴와 통일).

## 2. `/report` 페이지 — 레이아웃

- **상단**: 차트 페이지(`app/chart/page.tsx`)와 같은 패턴
  - **연·월 표시** (예: `2025년 3월`)
  - **좌측 화살표**: 이전 달
  - **우측 화살표**: 다음 달
- 나머지 영역은 아래 「콘텐츠 유무」에 따라 분기.

## 3. 콘텐츠 유무에 따른 UI

### 3-1. 해당 월에 보고서(또는 요약 콘텐츠)가 **없을 때**

- **「보고서 작성하기」** 버튼(또는 동일 의미의 주요 CTA)이 보이도록 한다.
- 사용자가 **작성하기**를 누르면:
  - **엑셀 추가 페이지**(`app/excel/page.tsx`)와 유사한 **로딩 UX**
    - 경과 시간 표시: **초 단위** (예: `걸린 시간: N초`, 1초마다 증가)
    - **「로딩중…」** 문구
    - **로딩 스피너** (엑셀 페이지와 시각적 톤 맞추기)
- 데이터 **조회·집계** 방식은 **§5** 참고. 생성된 보고서 **본문을 DB에 저장할지**는 **§5-3**·§3-2와 같이 별도 설계.

### 3-2. 해당 월에 콘텐츠가 **있을 때**

- (추후 정의) 예: 저장된 보고서 본문 표시, 수정 버튼 등.  
  → **이번 기획에서는 「없을 때」 플로우와 네비게이션만 명확히 한다.**

## 4. 엑셀 페이지와 맞출 UX 요약

| 항목         | 엑셀 추가하기 참고       | 보고서 작성하기  |
| ------------ | ------------------------ | ---------------- |
| 경과 시간    | 초 단위 증가             | 동일             |
| 로딩 문구    | 로딩중…                  | 동일             |
| 스피너       | 원형 스피너 등           | 동일 스타일 권장 |
| 완료 후 알림 | 인라인 문구 (alert 없음) | 추후 정의        |

## 5. 데이터 조회 방법 (보고서용)

이 프로젝트에서 **거래 원본은 MySQL**에만 있다 (`transactions` / `mytransactions`, Prisma 모델 `Transaction` / `MyTransaction`). 보고서는 **별도 캐시 파일 없이**, 생성 시점에 **DB에서 읽어 집계**하면 된다.

### 5-1. 차트와 같은 진입점을 쓸지

- **권장**: Server Action(예: `generateReport`) **안에서만** 조회한다.
- 이미 있는 **`getTransactions(monthKey, ledgerOwner)`** (`actions/transactions.ts`)를 재사용하면 된다.
  - **`monthKey`**: `YYYY-MM` — 차트·엑셀과 동일한 문자열.
  - **`ledgerOwner`**: 사이드바 토글과 동일 (`"appa"` | `"gilwoong"`) — 어느 테이블을 읽을지 결정.
- **이번 달** 거래: `getTransactions(monthKey, ledgerOwner)`.
- **전월** 거래: `prevMonthKey`를 계산한 뒤 `getTransactions(prevMonthKey, ledgerOwner)` 한 번 더 호출.  
  - `prevMonthKey`는 `monthKey`에서 한 달 빼서 `YYYY-MM`으로 맞추면, **날짜 경계가 `getTransactions` 내부와 동일**해져 프롬프트의 `prevMonthStr`·숫자 집계가 일치한다.

### 5-2. `stats` 객체로 가공

- 프롬프트 끝의 `${JSON.stringify(stats)}`에 넣을 구조는 문서·프롬프트 초안에 맞게 서버에서 만든다.
- **카테고리별 합계**: `lib/constants.ts`의 **`aggregateByCategory`** 를 그대로 쓸 수 있다 (차트와 동일한 `ChartDatum[]` 흐름).
- **가맹점 TOP**, **평일/주말**, **총액·환불** 등은 `getTransactions`로 받은 **행 배열**을 서버에서 순수 함수로 집계하면 된다 (새 파일 `lib/reportStats.ts` 등으로 분리 검토).
- **Claude에 넣기 전에만** 집계하면 되므로, `/report` 페이지가 클라이언트에서 거래를 미리 불러올 필요는 없다 (엑셀 업로드처럼 **버튼 → 서버가 DB 조회 + 프롬프트**).

### 5-3. 생성된 보고서 본문을 어디에 둘지

- **이번 문서 범위**: “데이터 **조회**”는 위와 같이 **기존 MySQL + `getTransactions` + 집계 함수**로 충분하다.
- 보고서 **텍스트를 DB에 저장**할지는 스키마 추가가 필요하다. 구현 시 `prisma/schema.prisma`에 모델을 두거나, 별도 테이블을 정의하는 결정은 **별도 작업**으로 한다 (§3-2·§8 참고).

- `app/report/page.tsx` — (신규) 본 페이지

## 7. 프롬프트 · Claude 호출 방식 (엑셀 내보내기와 동일한 패턴)

엑셀 파이프라인(`actions/processExcel.ts`)과 **같은 방식**으로 맞춘다.

| 단계 | 엑셀 (`processExcel`) | 보고서 (예정) |
| --- | --- | --- |
| 위치 | `actions/processExcel.ts` (또는 분리 파일) | `actions/generateReport.ts` 등 **Server Action** 전용 모듈 |
| 프롬프트 | 파일 상단 **템플릿 문자열 상수** (`CLAUDE_PROMPT` 등) | 동일: **`CLAUDE_FEEDBACK_PROMPT`** 또는 빌더 함수 |
| 동적 데이터 | 배치 텍스트를 템플릿 끝 `데이터:` 뒤에 이어 붙임 | **집계된 `stats` JSON**을 템플릿 끝 `데이터:` 뒤에 이어 붙임 (엑셀의 `batchText`와 같은 역할) |
| 호출 | `Anthropic` 클라이언트 `messages.create` | 동일 모델·패턴 재사용 검토 |
| 가계부 구분 | `owner: LedgerOwner`로 테이블 분기 | 동일하게 **`ledgerOwner`** 인자로 집계 쿼리만 분기 |
| 클라이언트 | `app/excel/page.tsx`에서 `formData` + `ledgerOwner` 전달 | `app/report/page.tsx`에서 **`monthKey`**(및 필요 시 `ledgerOwner`)만 전달 — 파일 업로드 없음 |

즉, **“상수 프롬프트 + 서버에서 만든 문자열/JSON을 뒤에 붙여 한 번에 user 메시지로 보낸다”**는 점이 엑셀과 같다.

### 7-1. `month` · `prevMonthStr` 넣는 방법

UI의 연·월 상태(`year`, `month`)로 다음을 만든다.

- **`monthKey`**: `YYYY-MM` (차트·거래 조회와 동일). 서버 액션 인자로 넘기기 좋음.
- **`month` (프롬프트용 표현)**: 사람이 읽기 좋은 문자열. 예: `` `${year}년 ${month}월` `` → `"2025년 3월"`.  
  - 프롬프트 템플릿 안에서는 **템플릿 리터럴로 삽입**: `` `... 사용자의 ${monthLabel} 가계부 ...` ``  
  - 또는 서버 액션 안에서만 `const monthLabel = formatMonthKR(monthKey)` 로 만들고 인자로 받지 않아도 됨. **단, Claude에 넣는 문장과 집계 범위가 항상 같은 달을 가리키도록** `monthKey` 하나를 기준으로 파생시키는 것을 권장.

- **`prevMonthStr`**: **직전 달**을 가리키는 **표시용 문자열**.  
  - `monthKey`에서 날짜를 파싱해 `Date`로 한 달 빼기 → 같은 형식으로 포맷.  
    예: `monthKey === "2025-03"` → `prevMonthStr === "2025년 2월"` (또는 `"2025-02"`처럼 짧게 통일해도 됨 — **프롬프트·집계 모두 동일 규칙**만 맞추면 됨).
  - 프롬프트 본문에서 전월 비교 문구에 쓰인다: `` `전월(${prevMonthStr})` `` 처럼 **템플릿 리터럴**로 넣거나, `buildReportPrompt({ monthKey, stats, ... })` 안에서 한 번만 조합.

**집계 범위와의 대응**

- `categoryTotals`·거래 집계는 **`monthKey`의 달**만 포함.
- `prevCategoryTotals`는 **`monthKey`의 이전 달**만 포함 (날짜 경계는 `getTransactions` / Prisma `where`와 동일한 규칙).

이렇게 하면 프롬프트 안의 `${month}` / `${prevMonthStr}` 문구와 실제 숫자 데이터가 어긋나지 않는다.

### 7-2. 사용할 프롬프트 본문 (초안)

아래는 **서버에서** `month`, `prevMonthStr`, `stats`를 주입하는 형태의 예시다. (엑셀처럼 상수 + 마지막에 `데이터:` 블록.)

```typescript
const CLAUDE_FEEDBACK_PROMPT = `
아래는 사용자의 ${month} 가계부 통계 데이터야 (SQL 집계 결과).

[사전 계산 - 이미 계산된 값이므로 그대로 사용]
- categoryTotals: 이번달 카테고리별 지출 합계
- prevCategoryTotals: 전월(${prevMonthStr}) 카테고리별 지출 합계
- weekdayStats: 평일/주말 일수 및 지출 합계
- topMerchants: 자주 방문한 가맹점 TOP 5
- totalStats: 총 지출 / 총 환불 금액

[분석 항목]

1. 📊 ${month} 요약
- 총 지출 금액과 환불 금액을 한 문장으로 자연스럽게 작성
- 카테고리별 지출을 금액과 비율로 나열 (높은 순)

2. 🏆 이달의 TOP
- 지출이 많은 카테고리 TOP 3를 금액과 비율로 언급
- 자주 방문한 가맹점 TOP 3를 방문 횟수와 총 금액으로 언급

3. 🥗 엥겔지수 분석
- 엥겔지수 = (식비 + 카페) / 총 지출 * 100
- 반드시 아래 형식으로 작성:
  "이번 달 식비·카페 합산 {식비+카페 금액}원으로 전체 지출의 {엥겔지수}%예요.
   카페를 격일로 줄이면 월 약 {카페금액/2}원,
   외식을 주 1회 줄이면 약 {식비/식비건수}원 절약할 수 있어요."
- 25% 미만이면: "식비·카페 관리가 잘 되고 있어요 👍" 한 문장으로 마무리
- "높아요", "줄이세요", "주의하세요" 같은 단순 경고 표현 절대 금지
- 수치 없는 코멘트 금지. 반드시 구체적인 금액 포함

4. 📅 주말 vs 평일 패턴
- 평일 1일 평균 = 평일 total / 평일 days
- 주말 1일 평균 = 주말 total / 주말 days
- 두 수치를 나란히 언급하고 자연스럽게 한 문장으로 마무리
- 주말 1일 평균이 평일의 2배 이상일 때만:
  "주말 지출이 평일의 {N}배예요. 주말에 {가장 많이 쓴 카테고리}가 집중되는 패턴이에요.
   주말 예산을 미리 {평일 1일 평균 * 1.5}원 정도로 정해두면 도움이 될 수 있어요."
- 2배 미만이면 비율 언급 없이 수치만 나열

5. 📈 전월 대비
- 전월 대비 50% 이상 증가한 카테고리만:
  "{카테고리} 지출이 지난달보다 {증가율}% 늘었어요. ({전월금액}원 → {이번달금액}원)"
- 해당 없으면: "이번 달은 전월 대비 급격히 늘어난 항목이 없어요 👍"
- 증가 이유를 단정짓는 표현 금지

6. 💡 총평 & 다음 달 조언
- 잘한 점 1가지 이상 반드시 구체적인 수치와 함께 언급
- 다음 달 실천 가능한 조언 1~2가지를 구체적인 금액과 횟수로 제시
- 마지막은 격려 한 문장으로 마무리

[절대 금지]
- 표, 마크다운 테이블, 코드블록 사용 금지
- "줄여야 합니다", "위험합니다", "문제입니다" 같은 부정적 단정 표현 금지
- 수치 없는 막연한 조언 금지 (ex. "카페를 줄여보세요" → 금지)
- 항목 순서 변경 금지

[응답 형식]
- 각 항목은 이모지 헤더로 시작
- 친근하고 실용적인 톤
- 전체 길이는 400~600자 이내

데이터:
${JSON.stringify(stats, null, 2)}
`;
```
